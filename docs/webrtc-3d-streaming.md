> 특정 구현체가 아니라, “구조”를 정리합니다.

### 이 문서가 다루는 것

- 웹 기반 실시간 3D 스트리밍에서 반드시 등장하는 컴포넌트
- 책임 분리(경계) 기준
- 세션 생명주기(생성~종료) 설계 포인트

---

## 1) 핵심 컴포넌트

- **Client**: 입력/화면 수신
- **Signalling**: 연결 협상(세션 시작을 위한 제어 채널)
- **Matchmaking**: “어떤 런타임에 붙을지” 결정
- **Runtime(Engine)**: 실제 렌더링/인코딩 수행
- **Session Store**: 세션 상태 기록(메모리/Redis/DB)

---

## 2) 설계의 핵심은 “경계”

스트리밍 시스템은 기능보다 **경계가 먼저**입니다.

- Signalling은 “협상”까지만
- Matchmaking은 “선택/라우팅”까지만
- Runtime은 “실행/리소스 소비”만
- Backend는 “정책/규칙/기록”을 담당

경계가 섞이기 시작하면 장애 원인이 추적되지 않고, 운영 난이도가 급격히 상승합니다.

---

## 3) 세션 생명주기 (요약)

1. 접속 요청
2. 매칭(런타임 선택)
3. 협상(Signalling)
4. 스트리밍 시작
5. 이탈 감지
6. 런타임 종료/유지 정책 적용

운영 관점에서 중요한 건 **4 이후**입니다.  
“유저가 떠났을 때 무엇을 정리할지”가 비용과 안정성을 결정합니다.
